# Microarchitecture Notes

## Cores

The repository keeps two CPU implementations:

- `rv32i_core.sv`: preserved single-cycle reference core.
- `rv32i_pipeline_core.sv`: 5-stage pipelined core used for pipeline, trace, randomized, and branch-prediction work.

## Pipeline Stages

The pipelined core uses the classic 5-stage structure:

- IF: instruction fetch and next-PC selection
- ID: instruction decode and register read
- EX: ALU, branch/jump target calculation, and branch resolution
- MEM: data memory access
- WB: register writeback

Pipeline registers carry instruction, PC, valid bit, control signals, operands, immediates, destination register, and writeback selection metadata.

## Control Flow

JAL and JALR resolve in EX and redirect fetch when taken.

BEQ and BNE now use the Phase 6 conditional branch predictor during IF. Branches still resolve in EX, where prediction correctness is checked. A misprediction redirects the PC and flushes younger wrong-path instructions.

## Branch Predictors

The Phase 6 predictor is a small 1-bit table:

- File: `rtl/branch_predictor.sv`
- Entries: 16
- Index: `PC[5:2]`
- Reset value: not taken
- Update: resolved conditional branch outcome in EX

The Phase 7 predictor is a small gshare predictor:

- File: `rtl/gshare_branch_predictor.sv`
- PHT entries: 16
- GHR width: 4 bits
- Counter type: 2-bit saturating
- Initial counter state: weakly not taken
- Index: `PC[5:2] XOR GHR`

`rv32i_pipeline_core` selects no prediction, simple prediction, or gshare using the `BP_MODE` parameter. Misprediction recovery remains the same: branch resolves in EX, the PC redirects to the correct path, and younger valid bits are flushed.

## Observability

The pipeline exposes debug outputs for:

- IF/ID/EX/MEM/WB valid, PC, and instruction
- WB destination, write enable, and write data
- stall, flush, redirect, and redirect target
- branch predictor predicted/actual direction and target
- gshare GHR and predictor index
- branch predictor counters

Trace logs and VCD waveforms are generated by the testbenches rather than by synthesizable logging logic.

## Phase 10 Memory-Latency Path

The pipelined core has a `DMEM_LATENCY_CYCLES` parameter for the data-memory path.

- `1` keeps the baseline one-cycle memory behavior.
- Values greater than `1` hold a valid LW/SW in MEM for extra wait cycles.
- While MEM is waiting, PC, IF/ID, ID/EX, and EX/MEM are held, and MEM/WB receives a bubble.
- Store write-enable is delayed until the wait completes.
- Load data is captured into WB only after the wait completes.

The core also has a conservative decode-stage RAW interlock. If the instruction in ID needs a register that an older in-flight instruction will write, the frontend is held and a bubble is inserted into EX until the producer has written back. This handles delayed-load dependencies but is not bypass forwarding.

Additional counters expose memory-stall behavior, including total memory stalls, load stalls, store stalls, and load-use/RAW load stalls.



## Phase 11 Direct-Mapped D-Cache

The pipelined core can now route LW/SW traffic through a small direct-mapped D-cache experiment using `DCACHE_ENABLE`.

- File: `rtl/direct_mapped_dcache.sv`
- Default cache lines: 4
- Line size: one 32-bit word
- Default index for four lines: address bits `[3:2]`
- Store policy: write-through, write-allocate
- Miss behavior: assert a memory stall for `DCACHE_MISS_PENALTY_CYCLES`

When the cache is disabled, the module preserves the Phase 10 memory-latency style path. When enabled, hits complete without the miss penalty and misses hold the pipeline using the existing MEM-stage stall mechanism.

I-cache is intentionally deferred. Instruction fetch still uses the simple instruction memory from earlier phases.

## Phase 12 Simplified L1/L2 D-Cache Path

The data-memory path now supports an optional simplified L2-style backing cache behind the Phase 11 L1 D-cache. The model remains direct-mapped and one word per line at both levels.

- L1: 4-line direct-mapped D-cache by default
- L2: 8-line direct-mapped backing cache by default
- Store policy: write-through, write-allocate
- L2 hit latency and L2 miss penalty are parameterized

This is a data-cache experiment only. Instruction fetch still uses the simple instruction memory. The model is intended for hit/miss and CPI experiments, not production cache hierarchy behavior.

## Phase 13 Simplified UCP-Style Partitioning

Phase 13 adds a trace-level shared-cache partitioning model rather than changing the RTL pipeline. This keeps the Phase 12 cache hierarchy stable while exploring cache-management policy ideas.

- Model file: `scripts/ucp_partition_model.py`
- Workload traces: `tests/benchmarks/ucp/`
- Total shared lines in the default experiment: 4
- Line size: 4 bytes
- Workload model: logical stream labels such as `A` and `B`
- Policies: static equal partition and deterministic utility-guided partition

The utility-guided policy evaluates legal allocations and chooses the allocation with the most total hits, then fewest misses, then closest-to-equal allocation. The reported penalty cycles are fixed-miss-penalty estimates from trace replay, not pipeline-integrated CPI.

This phase is preparation for future SMT or shared-cache experiments. It is not a production UCP controller and does not implement cache coherence or hardware thread IDs.

## Phase 13.5 RTL UCP Cache Experiment

Phase 13.5 adds a pipeline-integrated RTL cache hierarchy experiment:

```text
Pipeline MEM stage -> private L1 bank -> shared L2 -> UCP-partitioned L3 -> backing memory
```

The CPU still has one pipeline. Logical core/stream ID is derived from the data address. L1 is modeled as private per stream, L2 remains shared and unpartitioned, and UCP-style fixed partitioning is applied only to L3.

Reports are generated by `scripts/run_ucp_tests.ps1` at `reports/perf/ucp_rtl_partition_summary.md` and `.csv`.

## Phase 13.6 Dynamic UCP

Phase 13.6 adds a dynamic UCP monitor for the L3 experiment. Policy mode 2 uses shadow tag arrays to estimate utility for each two-stream L3 split, performs exhaustive split selection at a configurable interval, and safely invalidates L3 when the allocation changes. Private L1 and shared unpartitioned L2 behavior remain unchanged.

## Phase 15 SMT-Style Pipeline

Phase 15 adds a separate two-thread experimental pipeline rather than modifying the original single-thread pipeline. The SMT core has per-thread PC state, per-thread architectural registers, and a one-bit thread ID carried through every pipeline stage.

The fetch policy is deterministic round-robin. Data hazards are checked only within the same thread by comparing both register number and thread ID. Writeback writes only the selected thread's register bank. Branch and jump redirects update the resolving thread's PC; the current flush policy is conservative and global.

The cache hierarchy can now select UCP stream ID from either the legacy address split or the pipeline thread ID. In SMT mode, thread 0 maps to stream 0 and thread 1 maps to stream 1.

## Phase 16 Scoreboard / Readiness Model

Phase 16 adds a standalone `scoreboard_issue_model.sv` beside the CPU cores.
It tracks per-thread architectural-register busy state and producer tags, then
allocates reservation-station-like entries from `reservation_station_entry.sv`.
Each entry records source-ready bits and source tags. A thread-aware result
broadcast wakes matching operands and clears the matching register owner.

This is intentionally separate from architectural execution. It demonstrates
dependency tracking and readiness modeling while preserving in-order CPU
behavior. There is no reorder buffer, speculative commit, physical-register
renaming, execution-unit scheduling, or load/store queue.


## Phase 17 Tomasulo-Style Scheduling Experiment

Phase 17 adds a standalone `tomasulo_experiment_core.sv` rather than changing the main CPU pipeline. The model contains four reservation-station entries, a per-thread register-status table, one ALU execution slot, and a single CDB-style broadcast path.

The register-status table is a rename-lite mechanism: each architectural register can be marked busy with the tag of its current producer. Reservation stations either hold ready operand values or wait on source tags. When the CDB broadcasts a matching tag for the same thread, waiting operands wake up.

A younger ready instruction may issue before an older waiting instruction. Stale producer results are ignored if the destination register is now owned by a newer tag. This demonstrates Tomasulo-style dynamic scheduling concepts without adding a ROB, physical-register free list, LSQ, branch speculation, or precise out-of-order commit.

## Phase 18 ROB Experiment

The standalone ROB experiment uses a 4-entry circular ROB connected to the existing Tomasulo-style reservation-station model. Register-status entries point to ROB tags. Completed ALU results write into ROB entries through a CDB-style path. Only the ready ROB head can retire and update architectural register state.

Memory operations and branches remain outside this model because they require LSQ, speculation, checkpointing, and recovery mechanisms that are planned for future phases.

## Phase 19 LSQ Preparation

The microarchitecture now includes a standalone limited LSQ experiment at `rtl/tomasulo_rob_lsq_experiment_core.sv`. It is separate from the main in-order cores. The experiment demonstrates load/store queue concepts: memory-uop allocation, address readiness, store-data readiness, conservative load ordering behind older unresolved stores, and store commit through the ROB.

The model is intentionally conservative and does not implement speculative load issue, store-to-load forwarding, violation replay, or cache/UCP integration for speculative memory requests.

## Phase 20 Integrated OOO Experiment

Phase 20 adds `ooo_experiment_core.sv`, a standalone integrated OOO-concept core combining reservation stations, CDB wakeup, register-status tags, ROB in-order commit, and a limited LSQ. It supports ADDI, ADD, SUB, AND, OR, XOR, LW, and SW. Branch speculation, full physical renaming, precise exceptions, store-to-load forwarding, memory replay, and cache/UCP integration are intentionally excluded.

## Phase 21 Final Integrated Target

The final target combines prior concepts into one product-style top: two logical SMT threads, gshare prediction, internal ROB/RS/LSQ-style structures, private L1 banks by thread, shared pseudo-LRU L2/L3 cache state, and dynamic UCP-style L3 allocation. The implementation is intentionally compact and educational; earlier standalone modules remain useful for studying each concept in isolation.
